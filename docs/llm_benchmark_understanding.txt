1. WHAT THIS PROJECT IS
───────────────────────
An LLM benchmarking suite for Rust that measures inference and training
performance across multiple Rust ML frameworks and backends. Compares
Rust-native frameworks (Burn, Candle) against Python/PyTorch to evaluate
whether Rust is viable for LLM workloads.

Part of a broader research project (CSE 291Y / 291P) titled:
  "FuzzBurn: Verifying and Benchmarking the Performance of Rust-Native
   LLM Agents"

The project has three benchmark sections controlled by feature flags,
plus a Python comparison benchmark.


2. THE THREE BENCHMARK SECTIONS + PYTHON COMPARISON
────────────────────────────────────────────────────

  Section 1: Burn Backend Throughput Comparison  [STATUS: FIXED — commit 791f617]
  ─────────────────────────────────────────────
  Purpose:    Apples-to-apples comparison of Burn backends on identical work.
  What:       A custom GPT-style decoder-only transformer (model.rs) is built
              with random weights and run in forward-pass-only mode across:
                - NdArray (pure Rust, CPU)
                - WGPU (GPU via WebGPU/Metal)
                - LibTorch (PyTorch C++ bindings, optional via `tch` feature)
  Model sizes: "tiny" (vocab=512, hidden=128, 2 layers, 2 heads, seq=64)
               "small" (vocab=2048, hidden=256, 4 layers, 4 heads, seq=128)
  Methodology: Same GptConfig, same random input tensor shape, same warm-up
               (3 passes) + 50 iterations. Only the backend differs.
  Metrics:    Input tokens processed per second (forward-pass throughput),
              per-iteration latency p50/p95, estimated kernel launch overhead,
              estimated memory bandwidth (GB/s), peak memory (RSS).
  Enabled:    Always (no feature flag needed).

  FIXED:      GPU sync barriers added (B::sync(device) after model.forward()),
              memory measurement replaced with cross-platform sysinfo crate.
              Verified results: NdArray tiny 2.14ms/59K tok/s, WGPU tiny
              2.82ms/45K tok/s, WGPU small 4.63ms/27K tok/s (GPU wins on
              larger model). Non-zero p50 latencies and RSS on macOS.

  Section 2: Candle Pure-Rust LLM (Real Weights, Real Prompts)  [STATUS: WORKING]
  ────────────────────────────────────────────────────────────
  Purpose:    End-to-end autoregressive generation benchmark using a real
              quantized model, measuring the full inference pipeline.
  What:       Loads a GGUF-quantized model via HuggingFace Candle and runs
              greedy decoding (temperature=0) on a fixed set of 5 prompts
              (short-creative, short-factual, medium-code, medium-reasoning,
              long-summarise) defined in prompts.rs.
  Models:     TinyLlama-1.1B Q4_K_M (default, ~700MB download)
              Llama-3.2-1B-Instruct Q4_K_M (requires HF token)
              Phi-4 and Qwen2.5-0.5B (registered but not yet supported)
  Methodology: Same prompts, same max_new_tokens, greedy decoding. Model load
               time measured separately and excluded from throughput numbers.
  Metrics:    TTFT (Time To First Token / prefill latency),
              output tokens/sec (decode throughput),
              per-token latency p50/p95,
              peak memory usage (RSS in MB).
  Enabled:    `--features candle` (optionally `candle,metal` on macOS).

  NOTE:       FireKing noted some weird TinyLlama prompt responses — possibly
              related to the GGUF model source (ripped from unofficial repo
              since official TinyLlama didn't have GGUF).

  Collected data (Llama-3.2-1B Q4_K_M):
    short-creative:   TTFT 104ms,  63.52 tok/s
    short-factual:    TTFT 137ms,  60.00 tok/s
    medium-code:      TTFT 280ms,  52.13 tok/s
    medium-reasoning: TTFT 490ms,  53.47 tok/s
    long-summarise:   TTFT 1846ms, 47.45 tok/s

  Section 3: Training Benchmark with Burn TUI Dashboard  [STATUS: WORKING]
  ─────────────────────────────────────────────────────
  Purpose:    Measures training throughput (forward + backward + optimizer)
              across Burn backends with autodiff enabled.
  What:       Trains the same GPT model (and a GptTranspose variant) with
              Adam optimizer on random token data, using Burn's built-in
              SupervisedTraining framework and TUI dashboard (ratatui).
  Backends:   Autodiff<NdArray> (CPU), Autodiff<WGPU> (GPU),
              Autodiff<LibTorch> (optional).
  GptTranspose: A variant model that uses weight tying via transpose —
              logits = hidden * mix_weight^T + mix_weight, then matmul with
              token_embedding^T. Exercises the transpose + addition pattern
              that fuzzburn identified as crash-prone in Burn's autodiff.
  Methodology: Same GptConfig, same batch size, random weights + random tokens.
               3 epochs, 256 training items, 64 validation items.
  Metrics:    Live loss curve, items/sec, epoch progress (TUI),
              estimated kernel launches, bandwidth, avg step time.
  Enabled:    `--features train`.

  Python Comparison Benchmark  [STATUS: WORKING, on `merging` branch]
  ────────────────────────────
  Purpose:    Direct Python/PyTorch baseline for Section 1 comparison.
  What:       Identical GPT architecture (same tiny/small configs) implemented
              in PyTorch, run with same batch_size=4, seq_len=32, 50 iters.
  Location:   python_benchmark/benchmark.py + python_benchmark/model.py
  Device:     Auto-detects MPS (macOS) / CUDA / CPU.
  Methodology: Same model config, same random input, 10 warmup + 50 measured
               iterations with proper device sync (torch.mps.synchronize()).
  Metrics:    Avg time/iter, p50 latency, throughput (tok/s).

  Collected data (PyTorch MPS):
    tiny:  avg 1246.86µs, 102,657.68 tok/s
    small: avg 3121.67µs, 41,003.67 tok/s


3. FILE STRUCTURE
─────────────────
  src/
    main.rs           Orchestrator: runs all 3 sections, prints system info.
                      On merging branch: CliConfig + parse_cli() for --model
                      and --list-models flags. RuntimeConfig from CLI + env.
    model.rs          GPT-style decoder-only transformer (Burn Module).
                      GptConfig (tiny/small presets), Gpt<B>, GptTranspose<B>,
                      TransformerBlock<B>, Mlp<B>.
    benchmark.rs      BenchmarkStats (forward-pass metrics), GenerationStats
                      (autoregressive metrics), BenchmarkConfig (batch/seq/iters),
                      MemoryStats, compare_results(), compare_generation_results(),
                      benchmark_model() function, memory measurement utilities.
    candle_runner.rs  Candle GGUF model loader and greedy decode loop.
                      CandleModelKind registry, CandleRunner with load()/generate().
                      On merging branch: resolve_model(), print_available_models().
    prompts.rs        Fixed prompt set (5 prompts) for reproducible benchmarks.
                      BenchmarkPrompt struct, BENCHMARK_PROMPTS, WARMUP_PROMPT.
    training.rs       Training benchmark using Burn's SupervisedTraining.
                      RandomLMDataset, LMBatcher, TrainStep/InferenceStep impls
                      for Gpt and GptTranspose, run_training_benchmark().
  python_benchmark/   (on merging branch only)
    benchmark.py      PyTorch benchmark runner — mirrors Section 1 methodology.
    model.py          PyTorch GPT model — identical architecture to model.rs.
  benches/
    llm_benchmark.rs  Scaffold for criterion benchmarks (placeholder only).
  Cargo.toml          Features: tch, candle, metal, cuda, train.
                      Dependencies: burn 0.20.1, candle 0.8, hf-hub 0.3,
                      tokenizers 0.21, sysinfo, criterion (dev).

6. HOW THIS FITS INTO THE BROADER RESEARCH PROJECT
───────────────────────────────────────────────────
The broader project has three components:

  a) fuzz_core / ad_trait fuzzer (cse291y/) — Differential fuzzer for the
     ad_trait and evalexpr_jit autodiff libraries. Scalar-level AST
     expressions. Found bugs in fract, acosh derivatives, sqrt branching.
     This is the part already written up in the report (291p_report_latex.txt).

  b) fuzzburn (fuzzburn/) — Differential fuzzer for the Burn tensor library.
     2D tensor programs with SSA IR. Found 17 autograd crashes.
     NOT YET in the report — needs to be added.

  c) llm_benchmark (this repo) — Performance benchmarking of Burn and
     Candle frameworks on real LLM workloads, compared against PyTorch.
     NOT YET in the report — needs to be added.

The connection between fuzzburn and llm_benchmark:

  - SAME BURN VERSION: Both use burn 0.20.1.
  - SAME BACKENDS: NdArray, WGPU, LibTorch tested in both.
  - TRANSPOSE BUG REPRODUCTION: GptTranspose in training.rs exercises the
    exact autodiff code path where fuzzburn found crashes.
  - AUTODIFF STRESS: Section 3 runs full forward+backward+optimizer through
    the same autodiff machinery fuzzburn found 17 crashes in.
  - NARRATIVE: correctness (fuzzing) + performance (benchmarking) = complete
    picture of the Rust ML ecosystem.

8. ENVIRONMENT VARIABLES / CLI FLAGS
─────────────────────────────────────
  Env vars (original, on benchmarking branch):
    LLM_BENCH_SECTION3_ONLY       Run only training section (skip 1 & 2)
    LLM_BENCH_CANDLE_MODEL        Model selection: tinyllama|llama32|phi4|qwen|all
    LLM_BENCH_HF_TOKEN / HF_TOKEN HuggingFace token for gated models
    LLM_BENCH_KERNEL_LAUNCH_US    Per-kernel launch time estimate (default 5µs)
    LLM_BENCH_TRAIN_TRANSPOSE_ONLY Run only transpose-tied model in section 3
    CANDLE_GGUF_PATH              Override path to local GGUF file
    CANDLE_TOKENIZER_PATH         Override path to local tokenizer.json

  CLI flags (added on merging/section-1 branches):
    --model <key>                 Pick model for Section 2 (e.g. tinyllama)
    --list-models                 Print available models and exit
    LLM_BENCH_MODEL env var       Fallback for --model

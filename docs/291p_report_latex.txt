\begin{abstract}


As Large Language Model (LLM) agents are increasingly integrated into autonomous decision-making pipelines, the demand for memory-safe, low-latency, and numerically reliable inference engines has grown. While PyTorch remains the industry standard, the emerging Burn framework offers a Rust-native alternative promising superior performance and safety. However, the lack of rigorous verification for its underlying tensor operations remains a barrier to adoption for high-stakes agentic tasks. In this paper, we present FuzzBurn, a dual-purpose framework for the systematic evaluation of Burn-based LLM agents. First, we implement a differential fuzzer that validates the correctness of Burn’s tensor arithmetic and autograd engine against PyTorch to detect numerical divergences and computational graph bugs. Second, we provide a performance benchmark focusing on the end-to-end latency of agentic loops, measuring the overhead of Rust-based API calls and startup costs compared to traditional Python environments. Our results demonstrate that while Burn offers significant advantages in memory footprint and execution speed, differential fuzzing is essential to ensuring the logical consistency required for truly reliable agent reasoning.
\end{abstract}


\begin{IEEEkeywords}
Rust, Benchmarking, Fuzzing, Automatic Differentiation, LLM Agents, Safety, Deep Learning
\end{IEEEkeywords}


\section{Introduction}
The machine learning (ML) community currently uses Python to develop inference systems for real-time applications because of its extensive library support for numerical analysis and automatic differentiation (AD), which include popular deep learning (DL) libraries such as PyTorch and TensorFlow \cite{mishra2024high, singh2025, baydin2015}. Despite strong institutional support by ML developers, Python has limitations in high-performance computing (HPC) applications, ranging from slow execution, performance overhead from memory fragmentation, and restricted support for concurrency due to its Global Interpreter Lock (GIL)\cite{mishra2024high, singh2025}.


Rust is an ideal language for HPC applications because it prevents memory leaks with its ownership model, offers memory-safe concurrency with its borrow checker, and prevents unsafe memory access by tracking the lifetime of references \cite{mishra2024high}. Rust clearly has advantages over Python for developing inference systems but the ML community has been slow to embrace Rust in HPC because it does not have a well-supported AD library \cite{liang2025adtrait}.


The \adtrait and \evalexprjit crates are emerging AD libraries for Rust that offer forward mode and reverse mode AD optimizations for HPC applications but they have not been well-tested \cite{liang2025adtrait, range2025}. Previous research has used differential fuzzing to identify numerical bugs in several mature DL libraries, including PyTorch and TensorFlow \cite{yang2023fuzzing}.


We aim to identify bugs in \adtrait and \evalexprjit using differential fuzzing in order to improve AD support in Rust and encourage more ML developers to use Rust in HPC applications. We introduce the \fuzzcore crate in order to support our effort in identifying numerical bugs in \adtrait and \evalexprjit\cite{guiney2025}.


\section{Background}
\subsection{Automatic Differentiation}
AD is a technique for computing the derivative of a function in a program by decomposing the function into a composition of elementary operations with known derivatives \cite{baydin2015}. Repeated applications of the chain rule on the gradient of the decomposed function can then be used to calculate the derivative of the composition of elementary operations \cite{baydin2015}.


AD has been used by ML developers to calculate the derivatives of complex algorithms with control flow statements using constant computational complexity, which provides efficiency for HPC applications over other expensive numeric and symbolic differentiation methods \cite{baydin2015}. Improving the quality of existing AD libraries in Rust would enable ML developers to take advantage of AD while benefiting from Rust's fast execution speed, memory-safe concurrency, and ownership model that eliminates latency spikes in inference systems \cite{mishra2024high}.


\subsection{Forward Mode vs Reverse Mode}
AD libraries have two methods of computing the derivatives of functions: forward mode and reverse mode \cite{yang2023fuzzing}.


Forward mode AD computes the derivative of the entire function by evaluating the chain rule on each elementary operation from the decomposed function from input to output, with the limitation that it can only compute the derivative of one input on one forward pass \cite{baydin2015, yang2023fuzzing}. Forward mode AD is therefore efficient for functions with few inputs and many outputs ($f: \mathbb{R}^n \to \mathbb{R}^m$ where $n \ll m$) \cite{baydin2015, yang2023fuzzing}.


Reverse mode AD computes the derivative of the entire function in the opposite way as forward mode AD, evaluating the chain rule on each elementary operation from the decomposed function from output to input \cite{baydin2015}. Reverse mode AD computes the derivative of the entire function in two phases: a forward phase and a backward phase \cite{yang2023fuzzing}. The forward phase builds a dependency graph from function values and the backward phase calculates the derivative of a function from output to input \cite{baydin2015, yang2023fuzzing}.


Reverse mode AD has a similar limitation as forward mode AD because it may need multiple reverse passes to compute the derivative of a function \cite{yang2023fuzzing}. Reverse mode AD is therefore efficient for functions with many inputs and few outputs ($f: \mathbb{R}^n \to \mathbb{R}^m$ where $n \gg m$) \cite{baydin2015, yang2023fuzzing}.


Since forward mode AD and reverse mode AD both represent different optimizations for computing derivatives of functions using different accumulation orders of the chain rule on a decomposed function, they should produce the same result when used on the same function because matrix multiplication is an associative operation \cite{baydin2015, yang2023fuzzing}.


\subsection{Differential Fuzzing}
Fuzzing is a popular technique for detecting security vulnerabilities in software by feeding generated inputs to a program under test (PUT) that may be malformed, either semantically or syntactically \cite{manes2019}. A reference oracle is a program that determines whether a PUT has a bug or security violation \cite{manes2019}.


Fuzzing is not sufficient for detecting implementation bugs that fail to crash a program because researchers have to manually check whether or not the PUT produces a correct output \cite{barr2015}.


Differential testing is a technique that exposes bugs in programs by comparing two programs that implement the same specification and testing whether both programs produce the same output for the same input \cite{mcKeeman1998}. We can use differential testing with fuzzing by feeding the same malformed input to two different PUTs that implement the same specification, which we will call differential fuzzing \cite{mcKeeman1998, yang2023fuzzing}. We know an implementation bug exists when both PUTs return different results for the same input because each PUT acts as a reference oracle for the other PUT \cite{yang2023fuzzing}.


We can use fuzzing in Rust by using the \cargofuzz crate in a program and compiling it against Rust's nightly build \cite{cargofuzz2025}. \cargofuzz is a program that invokes the \libfuzzersys crate, a wrapper for LLVM's fuzzing library for C++ \cite{cargofuzz2025, libfuzzer2025}.


\subsection{Existing Approaches to AD in Rust}
Existing approaches to implementing AD in Rust include the \tchrs crate, which implements a Rust binding for LibTorch \cite{tchrs2025}. LibTorch itself implements a C++ binding for the PyTorch library, a mature DL library that supports AD \cite{tchrs2025, mishra2024high}.


The primary limitation of \tchrs is that it breaks Rust's safety guarantees because it relies on a foreign function interface (FFI) to a C++ library, which can potentially introduce undefined behavior (UB) in a Rust program that uses the crate \cite{rustreference2025}. Relying on the \tchrs crate for AD therefore eliminates the advantages of using Rust for HPC applications and inference systems, which means that a native AD library for Rust is needed to take advantage of its memory-safe guarantees \cite{rustreference2025}.


\subsection{\adtrait}
The \adtrait crate introduces AD in native Rust by implementing a trait that can accumulate the necessary information for computing the derivative of a function \cite{liang2025adtrait}. \adtrait supports both forward mode and reverse mode AD, which means that we can use differential fuzzing on the crate to identify implementation bugs in either optimization \cite{liang2025adtrait}. The forward and reverse mode ADs for \adtrait serve as two of our PUTs for \fuzzcore.


\subsection{\evalexprjit}
The \evalexprjit crate is a mathematical expression evaluator that uses just-in-time compilation (JIT) and AD to compute partial derivatives and Jacobian matrices \cite{range2025}. \evalexprjit serves as another PUT for \fuzzcore. 


\subsection{PyTorch as a Reference Oracle}
PyTorch is a mature DL library in Python that supports dynamic tensor compilations with support for AD and hardware acceleration while matching the performance of other fast DL libraries \cite{pytorch2019}.


PyTorch stands apart from other DL libraries by using define-by-run execution, which differentiates functions by directly running the computation as opposed to creating a static graph ahead of time \cite{paszke2017}. PyTorch's define-by-run-execution gives developers the ability to use arbitrary control-flow constructs in exchange for additional overhead because it needs to calculate derivatives during each pass of the AD compilation instead of using a static graph \cite{paszke2017}.


PyTorch's maturity and performance advantages over other static DL libraries makes it an ideal use as a reference oracle for our differential fuzzing of \adtrait and \evalexprjit \cite{pytorch2019, paszke2017, mishra2024high}. Prior research by Yang also used PyTorch as a reference oracle for differentially fuzzing DL libraries with AD support, validating our decision to use it for \fuzzcore \cite{yang2023fuzzing}.


For the purposes of \fuzzcore, we will assume that PyTorch gives the accepted value for an AD compilation so that we can determine which optimization modes in \adtrait have implementation bugs. Although prior research by Yang found PyTorch itself to have implementation bugs in its AD calculations, PyTorch's maturity and widespread use by the ML community makes it a trusted source for establishing a reference oracle for our project \cite{yang2023fuzzing, mishra2024high}.


\section{Methodology}


\subsection{The \fuzzcore crate}


We created a Rust crate called \fuzzcore in order to differentially fuzz the \adtrait and \evalexprjit crates. We organized our project into a crate so that we can differentially fuzz multiple AD crates in Rust with minimal code reuse.


We can differentially fuzz a new AD library by creating an entry-point program called a fuzz target that uses the \fuzzcore crate. Our \fuzzcore crate implements a simple pipeline:
\begin{enumerate}
\item The \cargofuzz crate generates a stream of random bytes.
\item Our input decoder converts the random bytes from \cargofuzz into $N$ 64-bit (f64) floating-point (FP) inputs.
\item We create an abstract syntax tree (AST) to generate a random mathematical function using the $N$ f64 FP inputs from our input decoder as arguments.
\item Our fuzzing harness computes the derivative of our random function using PyTorch.
\item Our reference oracle compare the output of our PUT to the output of our reference oracle.
\end{enumerate}


\subsection{Input Generation}
Our input decoder generates inputs by interpreting a prefix of the random byte stream from \cargofuzz as a sequence of floating-point values.


Given $N$ variables, the decoder consumes the initial $N \times 8$ bytes of the stream. It splits the sequence into 8-byte chunks and interprets each chunk as an IEEE 754 double-precision (\texttt{f64}) value. We construct an input vector $\mathbf{x} \in \mathbb{R}^N$ to evaluate the generated AST.


\subsection{AST Generation}


We use the \texttt{arbitrary} crate to automatically generate random functions. The core data structure that we use is the \texttt{Expr} enum, which represents a recursive AST composed of unary and binary operations that terminates with variables or constants. The definition of \texttt{Expr} enum can be seen in \autoref{fig:expr-enum}.


\begin{figure}[H]
\begin{lstlisting}[caption={Expr enum implementation}, label={fig:expr-enum}]
pub enum Expr<T> {
    Number(T, f64),
    Id(T, String),
    UnOp(T, Op1, Box<Expr<T>>),
    BinOp(T, Op2, Box<Expr<T>>, Box<Expr<T>>),
}
\end{lstlisting}
\end{figure}


The generation process consumes \cargofuzz's byte stream as an \texttt{Unstructured} source to make deterministic, probabilistic choices. We constrain this process using an \texttt{AstGenConfig} struct, which limits tree depth and operator selection. The generation algorithm follows three key phases:


\begin{enumerate}
    \item \textbf{Recursive Structure:} The generator builds nested \texttt{Expr} nodes until it reaches the maximum depth or probabilistically selects a terminal.
    \item \textbf{Variable Scoping:} We maintain a stack of active variables to ensure that the expression is well-formed. When generating a terminal, the algorithm probabilistically decides whether to reuse an existing variable from the stack or introduce a new one. This ensures that variables are introduced sequentially and can be reused with a realistic distribution.
    \item \textbf{Operator Selection:} We implement the \texttt{Arbitrary} trait for operation enums (\texttt{Op1}, \texttt{Op2}), which allows the fuzzing harness to use the input bytes on a range of mathematical functions.
\end{enumerate}


The final output is a \texttt{GeneratedExpr} struct, which contains the root of the AST, the encapsulated variable environment, and the input arity. Our structure gives our fuzzing harness the necessary metadata to execute the expression across different AD libraries.


\subsection{Fuzzing Harness}


We created a fuzzing harness to differentially test \adtrait and \evalexprjit using Rust's trait system. A fuzzing harness is a program 
that acts as an adapter between the fuzzing engine and the PUT \cite{babic2019fudge}. The fuzzing harness is responsible for 
initializing the testing environment and translating the fuzzer's raw input bytes into function calls to the PUT \cite{babic2019fudge}.


We define a polymorphic interface, \texttt{MainBackend}, which specifies the abstract operations required for differentiation, 
including trigonometric functions, basic arithmetic, and zero initialization. In \autoref{fig:main-backend}, the \texttt{MainBackend} trait enforces a contract that every AD library must use.


\begin{figure}[H]
\begin{lstlisting}[caption={MainBackend Trait Implementation}, label={fig:main-backend}]
pub trait MainBackend: Sized + Clone {
    fn from_f64(val: f64) -> Self;
    fn zero() -> Self;
    fn cos(self) -> Self;
    ...
}


pub fn evaluate<T: MainBackend, Tag>(
    expr: &Expr<Tag>,
    env: &Env<T>,
) -> Result<T, String> {
    match expr {
        Expr::Number(_, val) => Ok(T::from_f64(*val)),
        ...
    }
}
\end{lstlisting}
\end{figure}


The \rust{MainBackend} trait serves as the interface for all evaluation engines. 


By enforcing a standard contract of operations for every target library, we decouple the definition of a mathematical expression from 
its execution. Our system allows the fuzzing harness to traverse a single generated AST and dispatch operations to multiple backends at 
the same time, which include traits such as \rust{Calculator} and \rust{PyTorchComputable}.


Since our system can evaluate the same input against multiple AD libraries, we can ensure that any differences in the outputs are due to implementation bugs rather than different inputs.


\subsection{Defining Oracles}
We defined two types of oracles for \fuzzcore: a reference oracle and a metamorphic oracle \cite{manes2019, barr2015}.


Our metamorphic oracle compares both of \adtrait's optimization modes against each other \cite{barr2015}. By treating each AD mode optimization as the oracle for the other optimization, we can identify implementation bugs in \adtrait without needing to know the accepted value for the derivative of our random function \cite{baydin2015, yang2023fuzzing}.


Our reference oracle uses PyTorch as the accepted value for the derivative of our random function because it is well-tested and has performance advantages over other DL libraries \cite{pytorch2019, paszke2017}. We individually compared PyTorch's computation to each optimization mode in \adtrait and \evalexprjit's AD computation to identify bugs in each library. Our reference oracle helped us determine the root cause for the implementation bugs in \adtrait after discovering them with metamorphic testing.


\subsection{Defining Hybrid Error Tolerances}


Our oracles need to set an error tolerance threshold in order to determine whether our PUTs have implementation bugs \cite{harris2020}. Our oracles cannot use the \rust{assert_eq!} macro because PyTorch accumulates gradients in a non-deterministic order and FP arithmetic is non-associative by definition \cite{goldberg1991, paszke2017}. We therefore need a hybrid error tolerance model that determines a threshold for how much our oracle and PUT can differ before we can determine whether or not our PUT has an implementation bug \cite{harris2020}.


We defined the following error tolerance model for our oracles \cite{harris2020}:


\begin{equation*}
|y_{ref} - y| \le \epsilon_{abs} + \epsilon_{rel} \cdot |y_{ref}|
\end{equation*}


We set $\epsilon_{abs}$ to $10^{-12}$, which is the absolute error tolerance for an oracle. The absolute error tolerance is used  when FP numbers are close to $0$ because relative errors grow unbounded when FP numbers approach $0$ \cite{harris2020}.


We set $\epsilon_{rel}$ to $10^{-9}$, which is the relative error tolerance for an oracle. The relative error 
tolerance is used when both FP numbers are large because absolute errors grow unbounded when both FP numbers approach infinity \cite{harris2020}.


We defined our tolerances based on the theoretical limits of IEEE 754 double-precision arithmetic, selecting conservative bounds that allow for accumulation error while remaining stricter than standard ML libraries \cite{hingham2002, paszke2017, harris2020}.


\subsection{Fuzz targets}
We created three fuzz targets that use \fuzzcore to differentially fuzz \adtrait and \evalexprjit.


\begin{enumerate}
    \item \textbf{\fuzztargetone}: An initial prototype using reverse polish notation (RPN) to generate random functions. We deprecated this target in favor of using random AST generation to support more complex equations.
    \item \textbf{\fuzztargettwo}: The primary target that differentially fuzzes \adtrait using ASTs to generate random functions.
    \item \textbf{\fuzztargetthree}: A secondary target that differentially fuzzes \evalexprjit using ASTs to generate random functions.
\end{enumerate}


Our fuzz targets support the following command line arguments during runtime:


\begin{enumerate}
    \item \textbf{\fuzzmode:} Controls the failure handling strategy. The default \texttt{PanicOnFirstError} calls the \rust{panic!} macro to halt execution when our reference oracle discovers an implementation bug in an AD library. We can alternatively use \texttt{continuous} to ignore failures so that the user can find multiple implementation bugs.


    \item \textbf{\fuzzoracle:} Selects the reference oracle:
    \begin{itemize}
        \item \texttt{rev\_fwd}: Differentially fuzzes \adtrait using a metamorphic oracle.
        \item \texttt{rev\_gt} / \texttt{fwd\_gt}: Differentially fuzzes one of \adtrait's optimization modes against our reference oracle.
        \item \texttt{all} (Default): Differentially fuzzes \adtrait using both metamorphic and reference oracles.
    \end{itemize}


    \item \textbf{\fuzztests:} Specifies the batch size of random functions generated per input stream from \cargofuzz (Default: 1).
\end{enumerate}

\section{Evaluation}


\subsection{Results}
\subsubsection{General Observations}


The following examples highlight specific issues that we encountered when differentially fuzzing \adtrait:


\begin{equation}
f(x) = \frac{x}{x}
\end{equation}
\begin{equation}
f(x) = \cos\left(\frac{x}{x}\right)
\end{equation}


\begin{itemize}
\item \textbf{FP Error Accumulation}: Forward and reverse passes exhibit differing FP error accumulation due to how they traverse the computational graph.
\item \textbf{Undefined Behavior}: We discovered UB when evaluating $\cos(\frac{x}{x})$.
\end{itemize}


\subsubsection{Forward Mode Edge Cases}


We also identified several issues within \adtrait's \texttt{ForwardAD} mode:


\begin{itemize}
    \item \textbf{Incorrect derivatives ($\operatorname{acosh}$)}: The implementation of the derivative for $\operatorname{acosh}$ incorrectly decomposes $\frac{1}{\sqrt{(x^2-1)}}$ as  $\frac{1}{\sqrt{(x-1)}} \cdot \frac{1}{\sqrt{(x+1)}}$.
    \item \textbf{Logarithm Function Handling}:
    The derivative of the natural logarithm is defined as:
    $$ \frac{d}{dx}\ln(|x|) = \frac{1}{x} $$
    The derivative of $\ln$ is well-defined for $x \neq 0$ and should be able to accept negative values of $x$ as inputs. However, the derivative for the \texttt{log} function in \texttt{ForwardAD} mode does not handle negative inputs and assumes that $x>0$, resulting in $\texttt{NaN}$. We fixed this by accounting for the edge case where the \texttt{log} function has an undefined derivative.
    \item \textbf{Divergent Branching in Square Root Functions}:
    The \texttt{sqrt} function exhibits divergent branching with UB for negative inputs.
    The derivative of $\frac{d}{dx}(\sqrt{x})$ correctly results in $\texttt{NaN}$ for $x < 0$. However, when the derivative of this function evaluates to $0$, \texttt{ForwardAD} still calculates the derivative as $\texttt{NaN}$ and propagates the result during the forward pass. This behavior incorrectly resulted in derivatives being evaluated as $\texttt{NaN}$.
    
    For example, the derivative:
    $$ \frac{d}{dx}(\sqrt{-1} + x) = 1 $$
    should not result in $\texttt{NaN}$ despite containing a complex number in the original expression.
\end{itemize}


\subsubsection{\evalexprjit}


\begin{itemize}
\item \textbf{AST fix}: We identified a bug in \evalexprjit's implementation of \texttt{Expr::Sqrt}, which incorrectly calculated derivatives with the form:
\begin{equation*}
\frac{d}{dx}(\sqrt{x}) = \frac{1}{2\sqrt{x}}
\end{equation*}


The AST for the derivative of \texttt{Expr::Sqrt(expr)} did not include the constant multiplication $\frac{1}{2}$, leading to an incorrect AST construction before JIT optimization passes.
\end{itemize}


We have submitted bug reports to the maintainers of \evalexprjit and \adtrait for each issue that we found.


\subsection{Case Study 1: Competing fract implementations in \adtrait}


Our metamorphic testing of the \adtrait crate uncovered an implementation bug in the forward mode AD optimization for the $
\mathrm{fract}$ function.


The \adtrait crate does not use a single source of truth (SSOT) between the forward AD and reverse mode AD optimizations. 
Rather than organizing code that is common to both optimizations into a single crate, \adtrait re-implements the same 
mathematical functions several times across files: \texttt{adfn.rs}, \texttt{adf.rs}, \texttt{adr.rs}, and \texttt{adr2.rs}. 
Each of these files re-implement the same function twice for each AD mode optimization.


\begin{figure}[H]
\begin{lstlisting}[caption={Incorrect fract implementation for forward mode AD in ad\_trait.}, label={fig:incorrect-fract-fwd}]
#[inline]
fn fract(self) -> Self { 
    Self::new(self.value.fract(), [1.0; N]) 
}
\end{lstlisting}
\end{figure}


\autoref{fig:incorrect-fract-fwd} shows an incorrect implementation of the $\mathrm{fract}$ function in \texttt{adfn.rs}, a math crate for the forward mode AD optimization in \adtrait. The $\mathrm{fract}$ function is defined as the fractional part of a FP number \cite{NIST:DLMF}:
\begin{equation*}\mathrm{fract}(x) = x - \floor{x}\end{equation*}


The code in \autoref{fig:incorrect-fract-fwd} should return the derivative of the $\mathrm{fract}$ function, which is defined as \cite{NIST:DLMF}:\begin{equation*}
\frac{d}{dx}(\text{frac}(x)) = 1 \quad \text{for } x \notin \mathbb{Z}
\end{equation*}


In \texttt{adfn.rs}, the derivative of $\mathrm{fract}$ is hardcoded to be $1$, which ignores the chain rule when forward mode AD calculates the derivative of a composite functions involving fractions. We need to use the chain rule in order to correctly calculate the derivative of the $\mathrm{fract}$ function composed with another function \cite{NIST:DLMF}:
\begin{equation*}
y = \mathrm{fract}(u(x))
\end{equation*}
\begin{equation*}
\mathrm{\frac{dy}{dx}}=\mathrm{\frac{dy}{du}}\cdot \mathrm{\frac{du}{dx}}=1 \cdot\ \mathrm{\frac{du}{dx}}
\end{equation*}


\autoref{fig:correct-fract-rev} shows the correct implementation for the derivative of the $\mathrm{fract}$ function for the reverse mode AD optimization in \adtrait:


\begin{figure}[H]
\begin{lstlisting}[caption={Correct fract implementation for reverse mode AD in ad\_trait.}, label={fig:correct-fract-rev}]
NodeType::Fract => { 
    tiny_vec!([f64; 2] => 1.0) 
}
\end{lstlisting}
\end{figure}


We see that \autoref{fig:correct-fract-rev} correctly implements the derivative of the $\mathrm{fract}$ function by calculating the local gradient of the $\mathrm{fract}$ function and passing the calculation to other nodes.


\subsection{Case Study 2: Incorrect derivative of acosh function in \adtrait}


When examining the source code for the reverse mode AD for \adtrait, we discovered an implementation bug for the $\mathrm{acosh}$ function in \texttt{adr.rs}.


The $\mathrm{acosh}$ function is defined as the inverse hyperbolic cosine function \cite{NIST:DLMF}. The derivative of the acosh function is defined as \cite{NIST:DLMF}:
\begin{equation*}
\frac{d}{dx}(\mathrm{acosh}(x)) = \frac{1}{\sqrt{(x^2-1)}}
\end{equation*}


\autoref{fig:incorrect-acosh-rev} shows the implementation of the derivative of the $\mathrm{acosh}$ function in \texttt{adr.rs}:


\begin{figure}[H]
\begin{lstlisting}[caption={Incorrect derivative of the acosh function in ad\_trait.}, label={fig:incorrect-acosh-rev}]
NodeType::Acosh => {
    let lhs = parent_0.unwrap();
    tiny_vec!([f64; 2] => 1.0/(ComplexField::sqrt(lhs - 1.0)*ComplexField::sqrt(lhs + 1.0)) )
}
\end{lstlisting}
\end{figure}


In \autoref{fig:incorrect-acosh-rev}, \texttt{adr.rs} calculates the derivative of the $\mathrm{acosh}$ function by decomposing the function into a product of $\frac{1}{\sqrt{(x-1)}}$ and $\frac{1}{\sqrt{(x+1)}}$.


The implementation of the derivative of $\mathrm{acosh}$ function in \autoref{fig:incorrect-acosh-rev} is mathematically equivalent to the standard implementation of the derivative when $x > 1$ but leads to a sign error when $x < 1$ \cite{brown2008}. For example, if $x = -2$, then derivative of the $\mathrm{acosh}$ function should be:


\begin{equation*}
\frac{d}{dx}(\mathrm{acosh}(-2)) = \frac{1}{\sqrt{(-2)^2-1}} = \frac{1}{\sqrt{4-1}} =\frac{1}{\sqrt{3}}
\end{equation*}


\adtrait evaluates the derivative of $\mathrm{acosh}$ function incorrectly by flipping the sign of the result when $x < 0$ due to an implementation bug:


\begin{equation*}
\frac{d}{dx}(\mathrm{acosh}(-2)) = \frac{1}{\sqrt{-2-1}} \cdot \frac{1}{\sqrt{-2+1}} = -\frac{1}{\sqrt{3}}
\end{equation*}


There is a lack of consistency in the way that the developers compute terms with a square root. For example, the $\mathrm{asinh}$ function is defined as the inverse hyperbolic sin function. The derivative of the $\mathrm{asinh}$ function is defined as \cite{NIST:DLMF}:


\begin{equation*}
\frac{d}{dx}(\mathrm{asinh}(x)) = \frac{1}{\sqrt{(x^2+1)}}
\end{equation*}


In \autoref{fig:correct-asinh-rev}, the reverse mode AD implements the derivative of the $\mathrm{asinh}$ function without decomposing the $\sqrt{(x^2 + 1)}$ term:


\begin{figure}[H]
\begin{lstlisting}[caption={Standard implementation of the derivative of the asinh function in ad\_trait.}, label={fig:correct-asinh-rev}]
NodeType::Asinh => {
    let lhs = parent_0.unwrap();
    tiny_vec!([f64; 2] => 1.0/(lhs*lhs + 1.0).sqrt())
}
\end{lstlisting}
\end{figure}


The inconsistency in the way that the \adtrait crate computes functions with square roots indicates that the crate suffers from manual optimizations of mathematical functions. We have highlighted some of the benefits of differential fuzzing in this example.


\section{Conclusion}


We presented \texttt{fuzz\_core} in this article, a differential fuzzing framework intended to support our effort in identifying 
numerical bugs in Rust's AD libraries. By using PyTorch as a reference oracle and implementing a hybrid error tolerance model, we 
successfully identified several bugs in \adtrait, including incorrect implementations for the derivatives of $\mathrm{fract}$ and $
\mathrm{acosh}$. Our findings demonstrate that even languages with strong memory guarantees require metamorphic testing to reduce 
mathematical errors in HPC applications.


Our results highlight that AD libraries in Rust are not yet mature enough to challenge established DL libraries in Python, 
including PyTorch. Differential fuzzing is therefore a necessary tool to ensure that Rust's AD libraries are well-tested and 
implemented correctly.


\subsection{Future Work}
We identify three key avenues for extending this research:


\begin{itemize}
    \item \textbf{API-Level Fuzzing:} While our AST generation targets the internal logic of mathematical operators, future iterations 
    could adopt the byte-driven, property-based approach used by tools such as NablaFuzz \cite{yang2023fuzzing}. Property-based approaches would allow us to explore edge cases at the API boundary, testing how libraries handle tensor shapes, broadcasting rules, and invalid memory layouts.
    
    \item \textbf{Optimization Oracles:} Currently, \fuzzcore detects correctness bugs. We propose extending the metamorphic oracle to detect missed optimizations by comparing the execution traces of JIT-compiled graphs against hand-optimized kernels. Our work is especially relevant for libraries such as \evalexprjit that rely on runtime optimization passes \cite{range2025}.
    
    \item \textbf{Compiler~Integration:} As Rust's experimental \texttt{autodiff} feature matures, we aim to extend our backend to 
    target \texttt{Enzyme} \cite{autodiff2018, NEURIPS2020_9332c513}. Since \texttt{Enzyme} operates as an LLVM pass, applying 
    differential fuzzing at this level would allow us to verify the correctness of gradients generated directly by the compiler backend 
    and beyond library-level verification \cite{NEURIPS2020_9332c513}.
\end{itemize}
